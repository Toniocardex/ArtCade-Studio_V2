# Pre-Release Native Rules

ArtCade Studio is in pre-release. Nothing is locked yet, so every feature,
style, layout, and system change must be treated as product architecture, not
as a patch over an existing problem.

## Core Rule

All functionality, styles, layout structures, themes, and runtime/editor code
must be native to the project design system and architecture.

Do not use cosmetic overrides, hidden workarounds, duplicated structures, or
temporary patches to make a feature look or behave correct. If something is
wrong at the source, fix the source.

## Applies To

- Dark theme
- Light theme
- Editor layout and panels
- Canvas/editor chrome
- Logic Board UI and behavior
- Script editor integration
- Runtime/editor contracts
- Build and preview flow
- Tests, fixtures, and generated files
- Documentation and design-system tokens

## Required Approach

- Prefer design tokens, shared components, and existing architecture.
- Move repeated styling into the native theme/token system.
- Make dark and light themes two first-class variants of the same system.
- Change component structure when the structure is wrong.
- Keep behavior inside the correct store, hook, service, or runtime boundary.
- Update tests and docs when a public behavior or design contract changes.
- Remove unused or dead CSS/code during related work.
- Mark genuinely temporary code clearly and plan its removal.

## Forbidden Patterns

- Theme-specific CSS used only to mask bad base tokens.
- One-off overrides that fight component defaults.
- Duplicated layout structures for visual convenience.
- Hardcoded colors when a token should exist.
- Hardcoded dimensions when layout constants already exist.
- CSS selectors that depend on accidental DOM structure.
- Runtime/editor sync fixes that bypass the official contract.
- Generated-file edits by hand when a generator exists.
- Silent behavior changes without tests or clear verification.

## Review Checklist

Before considering a change complete:

- Does it belong to the native design system or architecture?
- Is the source of truth updated instead of patched around?
- Are dark and light themes both coherent?
- Are there unnecessary overrides that should become base tokens?
- Did the change avoid dead code and duplicated CSS?
- Did tests/build pass for the affected area?
- If generated files changed, were they regenerated by the correct script?

If the answer is uncertain, stop and fix the architecture rather than layering
another workaround.
